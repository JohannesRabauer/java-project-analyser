= Introduction and Goals

== Requirements Overview

The Java Project Analyser is a headless MCP server that deeply understands Java projects.
A developer starts it via Docker Compose, points an MCP-capable AI agent (Claude Code, Cursor, VS Code Copilot) at it, and queries rich structural and semantic knowledge about the codebase — class relationships, architecture patterns, potential bugs, and more.

The system combines deterministic AST parsing (JavaParser) with LLM-powered semantic analysis across an eight-phase pipeline.
Results are stored as a searchable knowledge graph and RAG embeddings in PostgreSQL (pgvector).
External AI agents retrieve pre-computed facts instantly or trigger on-demand LLM reasoning (10–30 s) — all via MCP JSON-RPC 2.0 over stdio or SSE transport.

There is no browser UI for driving the analysis.
The MCP tool `analyse_project` triggers the pipeline; all interaction happens through the MCP protocol.
A minimal Thymeleaf landing page at `localhost:8080` shows server health and copy-pasteable agent configuration snippets.

== Quality Goals

The top five quality goals that drive architectural decisions, in priority order.
Each maps to a named approach in <<Solution Strategy>> (Chapter 4).

[cols="1,2,3,2", options="header"]
|===
| ID | Quality Characteristic | Scenario (short) | Architectural Approach

| QG-1
| Protocol Conformance
| Any standard MCP client (Claude Code, Cursor) establishes a JSON-RPC 2.0 handshake, discovers tools via `tools/list`, and invokes them with validated argument schemas — conforming to MCP specification 2024-11-05.
| Strict JSON-RPC 2.0 transport adapters (stdio & SSE) in Quarkus; Jackson schema validation on tool inputs; standard error codes on malformed requests.

| QG-2
| Modifiability
| A developer switches the LLM provider from OpenAI to local Ollama by changing one configuration property (`analyser.pipeline.model-provider`) — no code changes, no recompilation.
| Hexagonal architecture with `LlmPort` and `EmbeddingPort`; LangChain4J multi-provider abstraction; three independent model slots (pipeline, query, embedding).

| QG-3
| Deployability
| A developer who clones the repository runs `docker compose --profile local up` and has the full system (PostgreSQL + pgvector, Ollama, analyser) operational in under 5 minutes.
| Docker Compose with named profiles; self-contained Quarkus container image; pgvector bundled in the PostgreSQL image; no manual database migrations required.

| QG-4
| Analysis Accuracy
| For a well-known project (e.g., Spring PetClinic), the analyser correctly identifies all framework technologies, the architectural style, and surfaces at least one real code quality concern.
| JavaParser AST extraction (deterministic structural facts); phase-by-phase LLM prompting with structured output schemas; RAG retrieval grounding for on-demand queries.

| QG-5
| Time Behaviour
| A project with 300 production classes completes the full eight-phase pipeline (including RAG indexing) within 15 minutes using a local Ollama model.
| Chunked batching; parallel LLM dispatch; phase result caching (interrupted analysis resumes from last completed phase); batch embedding.
|===
