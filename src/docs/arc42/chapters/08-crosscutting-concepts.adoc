= Crosscutting Concepts

== 8.1 Threat Model (STRIDE)

[cols="1,1,3,3", options="header"]
|===
| ID | Category | Threat | Asset at Risk

| T-001
| Information Disclosure
| Source code of the analysed project is transmitted to a cloud LLM provider (OpenAI) without the operator's explicit awareness.
| Proprietary or confidential source code inside the mounted workspace.

| T-002
| Tampering
| A local attacker modifies cached phase result JSON files to corrupt or falsify analysis results.
| Integrity of the analysis output served via MCP tools.

| T-003
| Denial of Service
| Repeated `analyse_project` calls or on-demand tool calls exhaust LLM API rate limits, blocking legitimate use.
| Availability of cloud LLM providers.

| T-004
| Elevation of Privilege
| A path-traversal or symlink attack in the project path parameter causes the application to read files outside the workspace.
| Host filesystem confidentiality.

| T-005
| Repudiation
| No record exists of which project was analysed, with which model, or when — making it impossible to audit or reproduce results.
| Audit integrity.

| T-006
| Spoofing
| An unauthorized process connects to the SSE endpoint and issues MCP commands (analyse arbitrary paths, exfiltrate results).
| Confidentiality and integrity of analysis data.
|===

== 8.2 Security

[cols="1,3,2", options="header"]
|===
| Threat | Mitigation | Implementation

| T-001
| Documentation clearly states that `openai` profile sends source code to third-party APIs. The Thymeleaf landing page shows the active profile prominently.
| Landing page template; README documentation.

| T-001
| The `local` Docker Compose profile keeps all data within the Docker network — no external calls for LLM inference.
| Ollama container; network isolation.

| T-002
| Cache directory is created with `700` permissions (owner-only). Phase result files are written atomically (write to `.tmp`, rename).
| `FilesystemCacheAdapter`

| T-003
| LLM Adapter enforces exponential-backoff retry (max 3 attempts) and a per-job circuit breaker; on-demand tools have a concurrency limit.
| `LangChain4J LLM Adapter`

| T-004
| `ProjectTreeLoader` resolves all file paths relative to the workspace root and rejects any path whose canonical form does not start with the root.
| `ProjectTreeLoader.guardPath()`

| T-005
| Every LLM call is logged with timestamp, model, and token count. Phase results include job ID and completion timestamp.
| `AnalysisJobRepository`, structured logging.

| T-006
| SSE endpoint binds to `0.0.0.0:8080` inside Docker only. In production, expose only via Docker port mapping to localhost. Stdio transport is inherently single-client.
| Docker Compose `ports: "8080:8080"` (localhost binding configurable).
|===

== 8.3 Test

Testing follows the pyramid.
Each test references the Use Case or Business Rule it exercises.

[cols="1,2,2,2", options="header"]
|===
| Level | Scope | Tools | Trace Target

| Unit
| Domain model (value objects, state-machine transitions), JavaParser AST extraction, graph analysis, static analysis rules.
| JUnit 5, AssertJ
| Phase outputs, model invariants

| Integration
| LangChain4J adapters against WireMock stubs; `RagRepository` against Testcontainers PostgreSQL+pgvector; `ProjectTreeLoader` against temp directory fixtures; `McpDispatcher` request/response.
| JUnit 5, Testcontainers, WireMock
| LLM port contract, RAG search, MCP protocol

| Component
| Full pipeline with WireMock LLM stub; verifies phase ordering, cache persistence, resume behaviour, and RAG indexing end-to-end.
| JUnit 5, Testcontainers, WireMock
| UC-001, UC-002 (analyse + resume)

| End-to-End
| MCP stdio transport: send JSON-RPC messages, verify tool responses. MCP SSE transport: connect, send, verify SSE events.
| JUnit 5, HTTP client
| MCP protocol conformance, tool taxonomy
|===

== 8.4 Observability

=== Logging

Quarkus structured logging (JSON output in production).
Every log entry includes MDC fields `jobId` and `phaseId` when a pipeline operation is active.

Key log events:

* `AnalysisStarted` — job ID, project source, model config
* `PhaseStarted` / `PhaseCompleted` / `PhaseFailed` — phase ID, duration
* `LlmCallAttempt` — attempt number, model, estimated token count
* `LlmCallResult` — HTTP status, actual token usage, duration ms
* `McpToolCalled` — tool name, job ID, response time

=== Metrics

Quarkus Micrometer exposes metrics at `/q/metrics`.

[cols="2,2", options="header"]
|===
| Metric | Description

| `analyser.phase.duration` (timer, tag: `phaseId`)
| Wall-clock time per phase execution.

| `analyser.llm.calls.total` (counter, tags: `phaseId`, `status`)
| Total LLM call attempts, labelled `success` / `retry` / `failed`.

| `analyser.llm.tokens.prompt` / `.completion` (distribution summary)
| Token usage per call — informs cost tracking.

| `analyser.mcp.tool.duration` (timer, tag: `toolName`)
| Response time per MCP tool invocation.

| `analyser.jobs.active` (gauge)
| Number of currently running analysis jobs.
|===

=== Health

Quarkus health endpoints at `/q/health`:

* `LlmProviderHealthCheck` — verifies the configured LLM provider responds.
* `DatabaseHealthCheck` — verifies PostgreSQL connectivity.
* `WorkspaceHealthCheck` — verifies configured workspace paths are readable.

== 8.5 Error Handling

[cols="2,3,2", options="header"]
|===
| Error Scenario | Handling Strategy | Reference

| Transient LLM API error (HTTP 429, 5xx, timeout)
| Exponential-backoff retry, max 3 attempts (1s / 2s / 4s). After exhaustion: phase → `FAILED`; job → `FAILED`; cache flushed up to last successful phase.
| BR-016

| LLM response fails schema validation
| One clarification retry with a schema-reminder suffix. If retry also fails: log the raw response; phase result stored as `DEGRADED`; pipeline continues.
| BR-009

| SIGTERM / SIGINT received mid-pipeline
| Quarkus shutdown hook flushes completed phase results to cache. Run is resumable on next start.
| UC-002

| Project path inaccessible
| `analyse_project` tool returns a structured MCP error (`isError: true`) with a clear message. Job is not created.
| T-004

| LLM provider unreachable at startup
| `LlmProviderHealthCheck` reports `DOWN`; health endpoint fails; Docker restart policy triggers.
| Health check

| MCP tool called before analysis complete
| Tool returns structured error: "Analysis in progress. Phase N of 8 completed. Retry after completion."
| Cold start behaviour (PRD)

| On-demand tool LLM call times out
| 30-second timeout per on-demand call. Returns error to the MCP client with an explanation. Does not affect other concurrent requests.
| NFR: Time Behaviour
|===
