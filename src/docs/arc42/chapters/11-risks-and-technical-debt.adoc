= Risks and Technical Debt

== Risks

Risks are ordered by priority (= probability × impact on a 1–3 scale).

[cols="1,3,1,1,1,3", options="header"]
|===
| ID | Risk | Probability | Impact | Priority | Mitigation

| R-001
| *LLM output quality varies across providers and models.*
Phase results for obscure codebases may contain uncertainty markers or degraded sections, reducing report usefulness.
| 3 (high)
| 3 (high)
| 9 (critical)
| Phase prompts include explicit uncertainty instructions (BR-009, 8.5).
Degraded phases are surfaced in the Vaadin UI.
End-to-end tests (8.3) validate output against known projects.
ADR-003 enables switching to a more capable model.

| R-002
| *Vaadin-Quarkus integration may lag behind Vaadin major releases.*
The `vaadin-quarkus-extension` is a community extension; breaking changes in Vaadin or Quarkus may delay upgrades.
| 2 (medium)
| 2 (medium)
| 4 (medium)
| Pin extension versions in `pom.xml`; track the extension release calendar.
Hexagonal architecture (ADR-004) isolates the UI adapter; a UI rewrite does not touch the domain.
Cross-reference: ADR-002.

| R-003
| *Context window exhaustion for monolithic projects.*
A project with deeply nested packages or very large classes may generate a phase context that exceeds the model's limit even after chunking.
| 2 (medium)
| 3 (high)
| 6 (high)
| Chunked batching (BR-005) with configurable batch size.
Hard truncation safety net in `LangChain4JLlmAdapter` (SUC-003).
Monitor `analyser.llm.tokens.prompt` metric (8.4) to detect near-limit runs.

| R-004
| *Local Ollama performance on CPU-only hosts is too slow.*
Without a GPU, large models (7B+) produce responses in 60–300 s per call, violating QG-4.
| 3 (high)
| 2 (medium)
| 6 (high)
| Document GPU passthrough in the Docker Compose file (nvidia `deploy.resources` block in Chapter 7).
Recommend smaller models (e.g., `qwen2.5-coder:3b`) for CPU-only runs.
QS-04 explicitly scopes the time budget to cloud APIs; local runs are best-effort.

| R-005
| *External LLM API rate limits block large analyses.*
Cloud providers impose per-minute and per-day token quotas; a 500-class project may hit them mid-analysis.
| 2 (medium)
| 2 (medium)
| 4 (medium)
| Exponential-backoff retry (BR-016, 8.5).
Per-job circuit breaker in `LangChain4JLlmAdapter`.
Cache ensures no completed-phase tokens are re-spent on resume (BR-002).

| R-006
| *PostgreSQL becomes a hard startup dependency for all profiles.*
If the PostgreSQL container fails its health check (corrupt data volume, port conflict), the application container never becomes ready.
| 1 (low)
| 2 (medium)
| 2 (low)
| Docker Compose `depends_on: postgres: condition: service_healthy` prevents the app from starting before Postgres is ready.
Flyway migration retries on transient connection failures.
Testcontainers provides an isolated PostgreSQL instance in integration tests so CI has no dependency on a running database.
Cross-reference: ADR-007.

| R-007
| *Phase 9 knowledge graph outgrows relational SQL as projects scale.*
A large modular monolith with thousands of classes and deep package hierarchies may make multi-hop graph traversal on the `knowledge_graph_node`/`edge` tables prohibitively slow.
| 2 (medium)
| 2 (medium)
| 4 (medium)
| JSONB indexing and a well-designed `nodes + edges` schema cover typical project sizes.
Monitor query plans on `knowledge_graph_edge` joins; add a GIN index on `properties` if needed.
When SQL performance proves insufficient, introduce `adapter.out.graph.Neo4jKnowledgeGraphAdapter` behind the existing port — PostgreSQL continues to serve all other entities unchanged (ADR-004, ADR-006, ADR-007).
|===
