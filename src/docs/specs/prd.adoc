== Overview

=== Purpose

The Java Project Analyser is an AI-assisted Model Context Protocol (MCP) server that ingests a Java codebase, drives a ten-phase LLM analysis pipeline, and persists that model as a RAG (Retrieval-Augmented Generation) knowledge base.
The system is built as an AI-first tool: its primary interfaces are MCP tools, resources, and prompts, enabling any MCP-compliant AI system (e.g., Cursor, Claude Desktop, Claude Code) to analyze codebases and leverage the RAG database to assist with developer queries.

=== Problem Statement

Understanding and working within unfamiliar Java codebases is a massive cognitive tax for both human developers and AI coding assistants.
While standard AI assistants have limited context windows and cannot easily navigate complex structural patterns, this system automates deep structural analysis through a background pipeline and indexes the resulting knowledge model into a PostgreSQL database.
By exposing this indexed model over the Model Context Protocol, external AI systems can instantly retrieve exact code contexts, module hierarchies, test coverage assessments, and api endpoints without consuming massive prompt tokens or hallucinating details.

=== Target Users

[cols="1,3", options="header"]
|===
| User Group | Context

| External AI Systems
| LLM-based autonomous coding agents, editors (e.g., Cursor, VS Code with Copilot, Claude Desktop) that need structured, low-latency, deep context about a Java project's architecture, dependencies, and risks.

| Software Architects (via AI)
| Auditing an unfamiliar system for compliance, security, or readability by instructing their AI agent to run codebase queries.

| Onboarding Developers (via AI)
| Navigating a newly assigned service, asking their IDE agent to locate feature implementation points or explain module hierarchies using the MCP tools.

| Technical Writers
| Creating architecture documentation by prompting their AI assistant to fetch structured phase reports from the MCP server's resources.
|===

=== System Classification

The Java Project Analyser is a **headless, multi-transport Model Context Protocol (MCP) Server** backed by a persistent RAG database (PostgreSQL + pgvector).
It supports:
- **Stdio Transport**: JSON-RPC over Standard Input/Output for local IDE extensions (Cursor, VS Code) and command-line AI tools.
- **SSE (Server-Sent Events) Transport**: HTTP-based JSON-RPC for remote clients and web interfaces.

A completely separate and decoupled part of the application is the **Vaadin MCP Client Demo**.
This is a standard browser-based chat application that connects to the MCP server over SSE. It uses LangChain4J's MCP client integration to let a human chat with an LLM, which dynamically uses the MCP server's tools to fetch codebase context.

== Analysis Pipeline

The analysis pipeline consists of ten sequential phases.
The pipeline can be triggered on any project path in the filesystem or a git repository URL via the MCP tool interface.
Phases 1–9 extract and synthesise knowledge from the source code.
Phase 10 embeds and persists every phase result into the PostgreSQL-backed RAG knowledge base.

=== Phase 1 — Application Purpose Analysis

==== Goal
Establish the high-level reason for the application's existence before examining any implementation detail.

==== Inputs
* Project root directory or repository URL.
* Build descriptor files (`pom.xml`, `build.gradle`, `settings.gradle`).
* Top-level `README` files.

==== Processing
The analyser shall:
. Extract project coordinates, description fields, and declared plugins from the build descriptor.
. Locate and read any `README.md` / `README.adoc` file.
. Identify the application's main entry point class.
. Submit the collected context to the LLM with a prompt focused on business purpose, not implementation.

==== Outputs
Executive Summary::
One- to three-sentence description of what the application does.
Business Purpose::
Why the system exists; the business problem it solves.
Main Functionalities::
Bullet list of major capabilities.
System Classification::
One of: Web Application, Backend Service, Batch Processor, Integration Platform, Library / Framework, Desktop Application, Mobile Backend, Other.

---

=== Phase 2 — Module Analysis
Examines multi-module structures, build projects, and core boundaries.

=== Phase 3 — Package Analysis
Maps package hierarchies, namespaces, and cohesion metrics.

=== Phase 4 — Class Analysis
Analyzes key classes, high-priority models, and interfaces.

=== Phase 5 — Dependencies & Technology Stack
Discovers imported libraries, frameworks, database drivers, and runtime requirements.

=== Phase 6 — Architectural Patterns & Styles
Identifies structural styles (e.g., Hexagonal, MVC, Layered, Event-Driven) and design pattern compliance.

=== Phase 7 — Test Suite & Coverage Assessment
Analyzes testing types (Unit, Integration, E2E), libraries, and code coverage gaps.

=== Phase 8 — External Interfaces & APIs
Maps exposed endpoints (REST, gRPC, SOAP) and consumed web services.

=== Phase 9 — System Knowledge Graph
Assembles containment hierarchies, class dependencies, and external mappings.

=== Phase 10 — RAG Knowledge Base Indexing

==== Goal
Chunk, embed, and store every phase result into PostgreSQL using pgvector, enabling semantic similarity queries.

==== Inputs
* Phase results (1–9).
* Knowledge Graph from Phase 9.

==== Processing
The indexer shall:
. Chunk each phase result into semantically coherent text segments (default 512 tokens with 64-token overlap).
. Request a vector embedding for each chunk from the configured embedding model (e.g., Ollama `nomic-embed-text` or OpenAI `text-embedding-3-small`).
. Persist each chunk, its embedding vector, phase ID, and job ID to the `rag_chunks` table in PostgreSQL.
. Mark the analysis job as `INDEXED` in the database.

== MCP Protocol Interface (The Core API)

The primary interface of the Java Project Analyser is the Model Context Protocol (MCP) JSON-RPC 2.0 API. This allows AI systems to interact with the application.

=== MCP Tools

The MCP server registers the following tools for AI clients to call:

[cols="1,2,3", options="header"]
|===
| Tool Name | Parameters | Description

| `analyse_project`
| `projectPath` (string), `gitUrl` (string, optional)
| Triggers the background 10-phase analysis on a local project directory or clones a Git repository. Returns the `jobId`.

| `get_analysis_status`
| `jobId` (string)
| Returns the current pipeline phase, status (PENDING, RUNNING, COMPLETED, INDEXED, FAILED), and progress percentage.

| `search_codebase`
| `jobId` (string), `query` (string)
| Runs a similarity search on pgvector using the query embedding. Returns the top-K relevant text chunks and phase metadata.

| `find_bugs`
| `jobId` (string), `context` (string)
| Queries the RAG database to identify code smells, testing gaps, and bugs in the codebase.

| `get_project_summary`
| `jobId` (string)
| Returns the high-level business purpose and functional summary extracted during Phase 1.

| `get_ascii_report`
| `jobId` (string)
| Retrieves the finished, compiled AsciiDoc report for the codebase.
|===

=== MCP Resources

The MCP server exposes static and dynamic resource URIs that AI clients can read:

* `mcp://analysis/jobs` - Returns a JSON array of all analysis jobs, paths, and status values.
* `mcp://analysis/job/{jobId}/summary` - Exposes the Phase 1 executive summary and purpose.
* `mcp://analysis/job/{jobId}/structure` - Exposes the project's build and package organization.
* `mcp://analysis/job/{jobId}/report` - Exposes the raw compiled AsciiDoc report.

=== MCP Prompts

The MCP server registers prompts that assist AI clients in formatting instructions:

* `review-codebase-risks` - Sets up a structured prompt for the AI to query the codebase and compile an architectural risk report.
* `suggest-feature-impl` - Prepares a structured prompt to guide the AI in designing new features conforming to the codebase's architecture and conventions.

== Vaadin MCP Demo Client

The Vaadin web application acts as a browser-based client that demonstrates how an AI system utilizes the MCP server.

* **Mode of Operation**: The Vaadin UI does *not* talk to the database or LLM directly. Instead, it embeds a LangChain4J chat assistant which is configured with an `McpClient` pointing to the Quarkus MCP SSE endpoint (`/mcp/sse`).
* **Chat Interface**: The user chats with the assistant. If the user asks "What is the purpose of this project?" or "Find any bugs", the LLM client uses tool-calling to execute `get_project_summary` or `search_codebase` against the MCP server, displaying the response in the chat thread.
* **Pipeline Panel**: A dashboard view that lists active jobs. When the user enters a path and clicks "Analyse", the client calls the `analyse_project` tool on the MCP server and displays live-push progress using Vaadin `@Push` notifications.

== Non-Functional Requirements

=== Performance
* The analysis pipeline shall complete within 12 minutes for a project with up to 500 production classes.
* Query searches (pgvector similarity search) shall return results in less than 2 seconds.
* SSE message throughput shall support multiple concurrent AI clients without message drop.

=== Protocol Conformance
* The server shall conform 100% to the Model Context Protocol specification v1.0.0.
* Tool inputs must have strict JSON Schema schemas, and malformed requests must receive standard JSON-RPC 2.0 error responses.

=== Resumability
* Intermediate phase results shall be cached so that an interrupted analysis resumes exactly where it failed.

== Constraints and Assumptions

* Stdio transport is meant for single-user local integrations (such as IDE extensions). Only one client can connect via Stdio per process instance.
* SSE transport supports multi-client scenarios and is exposed over HTTP (port 8080).
* The pgvector similarity threshold defaults to 0.75 and is configurable via application properties.
