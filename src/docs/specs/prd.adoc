== Overview

=== Purpose

The Java Project Analyser is a local MCP server that deeply understands Java projects.
A developer starts it via Docker Compose, points it at a project, and any MCP-capable agent (Claude Code, Cursor, VS Code Copilot) can then query rich structural and semantic knowledge about that codebase — without reading every file itself.

=== Problem Statement

AI coding assistants have limited context windows and cannot easily navigate complex Java codebases.
They re-read files on every question, miss architectural patterns, and hallucinate structural details.

This server solves the problem by performing a deep, one-time analysis of a Java project — combining AST parsing (JavaParser), static analysis, and LLM-powered assessment — then exposing the results as a queryable MCP interface.
External AI agents retrieve exact code contexts, class relationships, architecture overviews, and quality assessments without consuming massive prompt tokens.

=== Target Users

[cols="1,3", options="header"]
|===
| User Group | Context

| Developers with AI agents
| The primary user. A developer working in Claude Code, Cursor, or any MCP-capable IDE. They start the analyser, connect their agent, and ask questions about architecture, bugs, class purpose, or dependencies.

| Software Architects
| Auditing unfamiliar systems by instructing their AI agent to query the MCP server for structural assessments and risk areas.

| Onboarding Developers
| Navigating a newly assigned service. Their agent queries class purposes, related classes, and module boundaries from the MCP server.
|===

=== System Classification

The Java Project Analyser is a **headless, multi-transport Model Context Protocol (MCP) Server** backed by PostgreSQL + pgvector.

It supports:

- **Stdio Transport** (primary): JSON-RPC over Standard Input/Output. Claude Code and Cursor launch the server process directly and communicate via stdin/stdout.
- **SSE Transport** (secondary): HTTP-based JSON-RPC for remote clients and browser-based scenarios.

A Thymeleaf landing page at `localhost:8080` shows server status, analyzed projects, and copy-pasteable MCP configuration snippets for common agents.

== Analysis Pipeline

The analysis pipeline runs when a developer triggers `analyse_project` via MCP.
It combines deterministic AST parsing with LLM-powered semantic analysis.
Results are stored as both structured data (graph) and embedded vectors (RAG).

The pipeline can analyze multiple projects concurrently (each identified by a job ID).

=== Phase 1 — AST Parsing & Graph Construction

==== Goal
Parse all Java files into ASTs and build a structural knowledge graph.

==== Processing
. Parse every `.java` file using JavaParser.
. Extract: classes, interfaces, records, enums, methods, fields, annotations.
. Build a graph of relationships: inheritance, implementation, composition, dependency (imports), method calls.
. Extract package hierarchy and module structure (multi-module Maven/Gradle).

==== Outputs
- Class graph with relationships (depends-on, extends, implements, contains)
- Package hierarchy
- Module boundaries (from build descriptors)
- Method signatures and annotations (REST endpoints, DI annotations, etc.)

---

=== Phase 2 — Dependency & Technology Extraction

==== Goal
Identify the technology stack, external dependencies, and build structure.

==== Processing
. Parse `pom.xml` / `build.gradle` for declared dependencies.
. Detect frameworks from annotations and imports (Spring, Quarkus, Jakarta EE, etc.).
. Map exposed endpoints (JAX-RS, Spring MVC, gRPC service definitions).
. Identify database drivers, messaging systems, external service clients.

==== Outputs
- Technology stack inventory
- REST/gRPC endpoint map with HTTP methods, paths, and controller classes
- External dependency list with versions
- Build module structure

---

=== Phase 3 — Graph Analysis & Metrics

==== Goal
Compute structural quality metrics from the graph.

==== Processing
. Calculate package cohesion and coupling.
. Detect circular dependencies.
. Measure class complexity (method count, LOC, fan-in/fan-out).
. Identify architectural layers and dependency direction violations.

==== Outputs
- Cohesion/coupling scores per package
- Circular dependency chains
- Complexity hotspots
- Layer violation report

---

=== Phase 4 — LLM: Project & Module Summaries

==== Goal
Generate human-readable summaries of the project and its modules using LLM reasoning.

==== Processing
. Submit project README, build descriptor, and top-level class names to the LLM.
. For each module/package: submit the class names, key interfaces, and structural role to the LLM.
. Generate executive summary, business purpose, and system classification.

==== Outputs
- Executive summary (1-3 sentences)
- Business purpose
- System classification (Web App, Backend Service, Library, etc.)
- Per-module purpose summaries

---

=== Phase 5 — LLM: Class Purpose Summaries

==== Goal
Generate a purpose description for each significant class.

==== Processing
. For each class with >10 LOC: submit the class source, its relationships, and package context to the LLM.
. Generate a one-paragraph purpose description.
. Batch classes to minimize LLM round-trips.

==== Outputs
- Per-class purpose summaries stored in the knowledge graph

---

=== Phase 6 — Static Analysis

==== Goal
Detect code quality issues deterministically (no LLM needed).

==== Processing
. Run pattern-based checks on the AST: unused imports, empty catch blocks, overly complex methods, missing null checks, raw type usage.
. Detect common anti-patterns: God classes, feature envy, data clumps.
. Check naming conventions and Javadoc presence.

==== Outputs
- List of warnings with severity, file, line number, and description
- Categorized by: style, potential bug, anti-pattern, documentation gap

---

=== Phase 7 — LLM: Architecture Assessment

==== Goal
Identify architectural patterns and assess design quality using LLM reasoning.

==== Processing
. Submit the structural graph summary, layer analysis, and dependency patterns to the LLM.
. Ask the LLM to identify: architectural style, pattern compliance, design strengths, risk areas.

==== Outputs
- Identified architectural style (Hexagonal, Layered, MVC, Event-Driven, etc.)
- Pattern compliance assessment
- Architecture risk areas
- Suggested improvements

---

=== Phase 8 — RAG Embedding & Indexing

==== Goal
Chunk and embed all knowledge into pgvector for semantic retrieval.

==== Processing
. Chunk source files into semantically coherent segments (split on class/method boundaries, ~512 tokens with 64-token overlap).
. Chunk LLM-generated summaries and assessments.
. Generate vector embeddings using the configured embedding model (Ollama `nomic-embed-text` or OpenAI `text-embedding-3-small`).
. Persist all chunks with their embeddings, metadata (file path, class name, phase), and job ID.

==== Outputs
- All knowledge embedded and searchable via cosine similarity
- Job status transitions to `INDEXED`

== MCP Protocol Interface

The primary interface is MCP JSON-RPC 2.0. Tools are grouped into pre-computed (instant) and on-demand (LLM-powered, heavier).

=== Pre-Computed Tools (Instant Response)

[cols="1,2,3", options="header"]
|===
| Tool Name | Parameters | Description

| `analyse_project`
| `projectPath` (string) or `gitUrl` (string)
| Triggers the analysis pipeline. Returns `jobId`. Progress queryable via `get_analysis_status`.

| `get_analysis_status`
| `jobId` (string)
| Returns pipeline phase, status (PENDING, RUNNING, COMPLETED, INDEXED, FAILED), and progress percentage.

| `get_project_summary`
| `jobId` (string)
| Pre-computed executive summary, business purpose, and system classification.

| `get_architecture_overview`
| `jobId` (string)
| Pre-computed architectural style, layer structure, module boundaries, pattern compliance.

| `get_class_purpose`
| `jobId` (string), `className` (string)
| Pre-computed one-paragraph explanation of what the class does and why it exists.

| `get_module_purpose`
| `jobId` (string), `moduleName` (string)
| Pre-computed module responsibility summary.

| `get_related_classes`
| `jobId` (string), `className` (string)
| From the dependency graph: classes this class depends on, classes that depend on it, inheritance chain.

| `get_class_diagram`
| `jobId` (string), `packageName` (string)
| PlantUML class diagram generated from the AST graph for the specified package.

| `get_checkstyle_warnings`
| `jobId` (string), `scope` (string, optional: file/package/all)
| Pre-computed code style issues and anti-pattern detections.
|===

=== On-Demand Tools (LLM-Powered, Heavier)

[cols="1,2,3", options="header"]
|===
| Tool Name | Parameters | Description

| `search_codebase`
| `jobId` (string), `query` (string)
| Semantic similarity search over indexed code chunks. Returns top-K relevant code segments. Medium cost — embedding generation + vector query.

| `get_possible_bugs`
| `jobId` (string), `scope` (string, optional: className/packageName/all)
| LLM analyzes relevant code chunks for potential bugs, null safety issues, race conditions, resource leaks. Heavy — 10-30 seconds.

| `suggest_improvement`
| `jobId` (string), `className` (string)
| LLM suggests refactoring opportunities, design improvements, and clearer abstractions for a specific class. Heavy — 10-30 seconds.
|===

=== MCP Resources

[cols="1,3", options="header"]
|===
| URI | Description

| `mcp://analysis/jobs`
| JSON array of all analysis jobs with status.

| `mcp://analysis/job/{jobId}/summary`
| Phase 4 executive summary.

| `mcp://analysis/job/{jobId}/architecture`
| Phase 7 architecture assessment.

| `mcp://analysis/job/{jobId}/graph`
| Serialized class dependency graph (JSON).
|===

=== MCP Prompts

[cols="1,3", options="header"]
|===
| Name | Description

| `review-codebase-risks`
| Guides an AI client through querying architecture, warnings, and related classes to compile a risk report.

| `suggest-feature-impl`
| Guides an AI client through retrieving architecture context and class relationships before proposing feature implementation.

| `onboard-to-project`
| Guides an AI client through understanding the project step-by-step: summary → architecture → module purposes → key classes.
|===

=== Cold Start Behaviour

When analysis is incomplete, tools return a structured message:

[source,json]
----
{
  "content": [{"type": "text", "text": "Analysis in progress. Phase 3 of 8 completed. Estimated 4 minutes remaining. Retry after completion."}],
  "isError": true
}
----

This enables the consuming LLM to understand the situation and retry.

== Model Configuration

The server supports separate model configuration for three concerns:

[source,properties]
----
# Pipeline model (bulk analysis — can be cheap/local)
analyser.pipeline.model-provider=ollama
analyser.pipeline.model-name=llama3

# Query model (on-demand reasoning — can be high-quality)
analyser.query.model-provider=openai
analyser.query.model-name=gpt-4o

# Embedding model
analyser.embedding.provider=ollama
analyser.embedding.model-name=nomic-embed-text
----

The developer chooses the cost/quality tradeoff per concern.
Using the same model for all three is valid and requires no extra configuration.

== Non-Functional Requirements

=== Performance
* The analysis pipeline shall complete within 15 minutes for a project with up to 300 production classes using a local Ollama model.
* Pre-computed tool responses shall return in less than 500ms.
* RAG search shall return results in less than 2 seconds.
* On-demand LLM tools shall return within 30 seconds.

=== Protocol Conformance
* The server shall conform to the Model Context Protocol specification (2024-11-05).
* Tool inputs have strict JSON Schema definitions.
* Malformed requests receive standard JSON-RPC 2.0 error responses.

=== Resumability
* Intermediate phase results are cached. An interrupted analysis resumes from the last completed phase.

=== Scalability
* The server handles multiple concurrent projects.
* Each project analysis runs on a managed thread.
* On-demand LLM tools handle concurrent requests without blocking each other.

---

== Constraints and Assumptions

* Stdio transport supports one client connection per process (matches how IDE extensions work).
* SSE transport supports multiple concurrent clients over HTTP (port 8080).
* pgvector similarity threshold defaults to 0.5 and is configurable.
* JavaParser is used for AST extraction — supports Java 8 through 21.
* Projects must use Maven or Gradle as build tool.
* Docker Compose is the supported deployment method (PostgreSQL + Ollama + app).

---

== Out of Scope for v1

* File watcher / automatic re-indexing on code change (manual re-trigger only)
* Anthropic model provider (Ollama and OpenAI only)
* Non-Java language support
* Native image compilation
* Graph database (pgvector + relational schema serves the graph needs for now)
