== Purpose

This system use case specifies the contract between the analysis pipeline and the LLM API.
It is not user-facing; it governs every call the pipeline makes to the language model.

== Input

[cols="1,1,3", options="header"]
|===
| Field | Type | Description

| `phaseId`
| integer (1–9)
| Identifies which analysis phase is being executed.

| `phaseContext`
| structured JSON
| The collected project data relevant to this phase (e.g., package listing for Phase 3, class metadata for Phase 4).

| `systemPrompt`
| string
| The phase-specific instruction template that tells the LLM what to analyse and what structure to return.

| `modelConfig`
| object
| `{ modelId, maxTokens, temperature }`.
Temperature is fixed at 0.2 for all analysis phases to reduce hallucination.

| `chunkIndex`
| integer (optional)
| Present when the phase context has been split into chunks.
Identifies the position of this chunk within the full context.

| `totalChunks`
| integer (optional)
| Total number of chunks for this phase.
|===

== Context Window Validation

Before constructing the request:

. Estimate the token count of `systemPrompt` + serialised `phaseContext` using the model's tokenizer.
. If the estimate exceeds the model's effective context window (90% of declared maximum, to leave headroom for the response):
  .. Split `phaseContext` into chunks no larger than `floor(0.6 × contextWindow)` tokens each.
  .. Execute one LLM call per chunk, passing `chunkIndex` and `totalChunks`.
. If a single chunk still exceeds the limit, log a warning and truncate to the maximum length, appending a truncation marker.

This implements <<../business-rules.adoc#BR-004,BR-004>>.

== Processing

=== Request Construction

The prompt sent to the LLM has three parts, in order:

. *System section*: the phase `systemPrompt` (constant per phase, cached for prompt-caching benefit).
. *Context section*: the serialised `phaseContext` (variable per call).
. *Output instruction*: a fixed instruction to return valid JSON matching the phase's output schema, to express uncertainty as `"unknown"` rather than guessing, and to use the exact field names defined in the schema.

=== API Call

. Submit the constructed prompt to the configured LLM API endpoint.
. Apply a per-call timeout of 120 seconds.
. On success (HTTP 200), proceed to response parsing.
. On transient failure (HTTP 429, HTTP 5xx, network timeout), apply the retry policy.

=== Retry Policy (<<../business-rules.adoc#BR-016,BR-016>>)

[cols="1,2", options="header"]
|===
| Attempt | Wait before attempt

| 1st retry | 1 second
| 2nd retry | 2 seconds
| 3rd retry | 4 seconds
|===

After three failed retries, raise a `PhaseExecutionException` with the last HTTP status and error body.

=== Response Parsing

. Parse the response body as JSON.
. Validate against the phase output schema.
. If validation fails:
  .. Retry once with a clarifying prompt appended: `"Your previous response was not valid JSON matching the required schema. Return only valid JSON with no explanation."`
  .. If the retry also fails, return a `DegradedPhaseResult` containing the raw response text and a `degraded: true` marker.

=== Success

[source,json]
----
{
  "phaseId": 3,
  "status": "COMPLETED",
  "chunkIndex": null,
  "totalChunks": null,
  "result": { /* phase-specific structured content */ },
  "llmCall": {
    "model": "claude-sonnet-4-6",
    "promptTokens": 4210,
    "completionTokens": 1380,
    "durationMs": 8340,
    "timestamp": "2026-05-26T14:22:10Z"
  }

----

=== Degraded Result

[source,json]
----
{
  "phaseId": 3,
  "status": "DEGRADED",
  "degradedReason": "Response did not match output schema after retry",
  "rawResponse": "...",
  "llmCall": { /* ... */ }
}
----

=== Phase Failed

Raises `PhaseExecutionException` (not returned as JSON).
The caller (pipeline) catches this, writes the failure state to cache, and terminates.

== LLM Call Log Entry (<<../business-rules.adoc#BR-007,BR-007>>)

Every API call — including retries and clarification retries — appends one entry to `<cache-dir>/llm-calls.jsonl`:

[source,json]
----
{
  "callId": "uuid-v4",
  "phaseId": 3,
  "chunkIndex": null,
  "model": "claude-sonnet-4-6",
  "attemptNumber": 1,
  "promptText": "...",
  "responseText": "...",
  "promptTokens": 4210,
  "completionTokens": 1380,
  "httpStatus": 200,
  "durationMs": 8340,
  "timestamp": "2026-05-26T14:22:10Z"
}
----

== EARS Requirements

When an LLM API call returns HTTP 429 or a 5xx status, the system shall wait the prescribed backoff duration before retrying.

While executing a chunked phase, the system shall merge all chunk results into a single phase result before writing to the cache.

If the LLM response does not match the expected output schema, the system shall retry exactly once with a schema-clarification prompt before recording a degraded result.

The system shall log every LLM call attempt, including failed retries, to the JSONL call log.
